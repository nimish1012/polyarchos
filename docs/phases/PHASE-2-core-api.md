# Phase 2 — Core API Service (Rust gRPC + REST)

**Date:** 2026-03-26
**Status:** Complete

---

## Objective

Build `services/core-api` as a production-grade dual-server Rust binary exposing:
- **gRPC** on `:50051` via tonic — implements `ComponentService` from proto contracts
- **REST** on `:8080` via axum — full OpenAPI 3 spec auto-generated by utoipa

Both servers run concurrently in a single process, sharing state via `Arc<RwLock>`.

### Source

| File | Purpose |
|---|---|
| `services/core-api/src/config.rs` | Reads `CORE_API_GRPC_PORT` and `CORE_API_REST_PORT` from env; falls back to `:50051` / `:8080`. |
| `services/core-api/src/error.rs` | Unified `AppError` type. Implements `IntoResponse` (axum) and `Into<tonic::Status>` (gRPC) so handlers are error-type-agnostic. |
| `services/core-api/src/store/mod.rs` | Thread-safe in-memory `ComponentStore` backed by `Arc<RwLock<HashMap>>`. Includes `seed_fixtures()` for local dev (3 synthetic AUTOSAR SWCs). Designed to be swapped for Neo4j + Milvus in Phase 4. |
| `services/core-api/src/grpc/mod.rs` | Includes tonic-generated proto stubs via `tonic::include_proto!`. |
| `services/core-api/src/grpc/component_service.rs` | tonic `ComponentService` implementation. Maps proto types ↔ domain types at the service boundary. Implements Get, List, Search (stub), Delete. |
| `services/core-api/src/rest/types.rs` | REST-specific request/response types with `serde` + `utoipa::ToSchema` + `IntoParams` derives. Separate from both proto and domain types. |
| `services/core-api/src/rest/handlers.rs` | axum handlers with `#[utoipa::path]` annotations. Implements GET `/api/v1/components/{id}`, GET `/api/v1/components`, POST `/api/v1/components/search`, DELETE `/api/v1/components/{id}`. |
| `services/core-api/src/rest/mod.rs` | Builds the axum `Router`. Declares the `#[derive(OpenApi)]` spec. Mounts Swagger UI at `/swagger-ui/` and the raw spec at `/api-docs/openapi.json`. |
| `services/core-api/src/main.rs` | Entry point. Initialises JSON structured logging, loads config, seeds the store, then starts both servers with `tokio::try_join!`. |

## Architecture Decisions

- **Proto-generated types do not leak** past the gRPC handler boundary. Domain types (`crates/domain`) are the internal currency; proto types are mapped at `grpc/component_service.rs`.
- **REST types are independent** of both proto and domain types. Defined in `rest/types.rs`, mapped in handlers. This allows the REST surface to evolve independently.
- **`tokio::try_join!`** — if either server crashes, the whole process exits. No silent half-broken state.
- **`protoc-bin-vendored`** — removes the system `protoc` requirement. Works on Windows, Linux, macOS without manual tool installation.
